시스템 아키텍처 STRUCTURE

의존성이 한 방향으로만 흐르도록 Include.h에서 모듈을 5계층으로 명시적으로 분리했습니다. 하위 계층(코어 도구)은 상위 계층(게임 로직)을 모르고, 상위가 하위를 사용합니다.

// Include.h — 의존성 레이어링 (실제 #include 순서가 곧 아키텍처)
// 1단계: 코어 도구     Singleton · Camera · Key · Sound · Collider
// 2단계: 게임 엔티티   Enemy · Boss · Knight
// 3단계: 매니저 시스템 MapManager · UIManager · TimeManager · EffectManager
// 4단계: 최상위 루프   GameManager · CMng
// 5단계: 씬 & 메인     Logo · Menu · Game · Over · Ending → Main

WinMain

게임 루프 — Fixed Timestep ENGINE

게임 엔진의 심장. PC 사양에 따라 게임 속도가 달라지지 않도록, 물리/로직은 고정 60틱으로 돌리고 렌더링은 가능한 만큼 빠르게 돌리는 구조를 직접 구현했습니다.

📄 Main.cpp — WinMain 메인 루프
const int TICKS_PER_SECOND = 60;
const int SKIP_TICKS = 1000 / TICKS_PER_SECOND;  // 한 틱당 16.6ms
const int MAX_FRAMESKIP = 5;

while (msg.message != WM_QUIT) {
  // 1) 메시지는 논블로킹으로 모두 비움 (게임이 멈추지 않게)
  while (PeekMessage(&msg, NULL, 0, 0, PM_REMOVE)) { ...
      g_Mng.chap[g_Mng.n_Chap]->OnMessage(&msg); }

  // 2) 고정 타임스텝: 밀린 틱만큼 Update 를 따라잡되, 최대 5번까지만
  while (GetTickCount64() > next_game_tick && loops < MAX_FRAMESKIP) {
    interpolation = float(GetTickCount64() + SKIP_TICKS - next_game_tick) / SKIP_TICKS;
    if (Gmanager.m_Pause == false)
      g_Mng.chap[g_Mng.n_Chap]->Update(interpolation);
    next_game_tick += SKIP_TICKS;  loops++;
  }
  // 3) 렌더링은 매 루프마다 (프레임은 가능한 만큼)
  Device9->BeginScene();  Device9->Clear(...);
  g_Mng.chap[g_Mng.n_Chap]->Draw();
  Device9->EndScene();  Device9->Present(...);
}

① PeekMessage

GetMessage(블로킹)가 아닌 PeekMessage로 입력을 비워, 입력이 없어도 게임이 계속 진행됩니다.

② Frame Skip

처리가 밀리면 Update를 여러 번 따라잡되 MAX_FRAMESKIP=5로 상한을 둬 "죽음의 나선" 방지.

③ Interpolation

틱 사이 보간값을 계산해 렌더 단계에 넘겨 부드러운 화면을 확보.

왜 중요한가 ▸ "왜 게임 루프에서 로직과 렌더를 분리하나요?"는 단골 질문입니다. 가변 델타타임의 물리 불안정성과 저사양에서의 게임 속도 저하를 동시에 막는 정석 구조라고 답할 수 있습니다.

CRTP 싱글톤 베이스 ENGINE

초기에는 매니저마다 전역 포인터 + new 또는 함수 내 static이 제각각이었습니다. 이를 CRTP(Curiously Recurring Template Pattern) 기반 Meyers 싱글톤 하나로 통일했습니다.

📄 Singleton.h
template <class T>
class Singleton {
public:
  static T* GetInstance() {
    static T instance;   // 최초 호출 시 1회 생성, 종료 시 자동 소멸
    return &instance;
  }
  Singleton(const Singleton&)            = delete;  // 복사 금지
  Singleton& operator=(const Singleton&) = delete;
protected:
  Singleton()  = default;
  ~Singleton() = default;
};
// 사용:  class Camera : public Singleton<Camera> { friend class Singleton<Camera>; ... };
//        #define CAM Camera::GetInstance()   →   CAM->Update();

이 설계가 해결한 것

  • Static init order fiasco — 전역 객체 간 초기화 순서 의존 제거 (지연 초기화)
  • 메모리 누수 — new 없이 지역 static → 프로그램 종료 시 자동 소멸
  • 스레드 안전 — C++11 이후 함수 내 static의 최초 1회 초기화는 표준 보장
  • 일관성 — Camera/Sound/TimeManager/EffectManager가 동일 규약 사용

호환을 위한 선택

기존 호출부가 CAM->, SOUND->처럼 포인터(->)를 쓰고 있어, 리팩터링 비용을 줄이려고 참조가 아닌 포인터를 반환하도록 의도적으로 설계했습니다. 대규모 코드베이스에서 인터페이스 호환을 지키는 실전 감각을 보여주는 대목입니다.

씬 관리 — 추상 인터페이스 + 다형성 ENGINE

모든 씬은 Chap 순수 가상 인터페이스를 구현하고, unique_ptr<Chap> 배열로 소유됩니다. 수명 관리를 스마트 포인터에 위임해 수동 delete를 제거했습니다.

리스크 (만약 raw 였다면)
Chap* chap[TOTALCHAP];
// new Logo(); ... 어딘가서 delete 누락 가능
// 기반 포인터 delete 시 ~파생() 미호출 → 누수/UB
현재 구현
std::unique_ptr<Chap> chap[TOTALCHAP];
chap[GAME] = std::make_unique<Game>();
// 종료 시 자동 해제
class Chap { virtual ~Chap() = default; ... }; // ★가상 소멸자
📄 Chap.h — 모든 씬의 공통 계약
class Chap {
public:
  virtual ~Chap() = default;          // ★ 기반 포인터 delete 안전 보장
  virtual void Init() = 0;
  virtual void Update(double frame) = 0;
  virtual void Draw() = 0;
  virtual void OnMessage(MSG* msg) = 0;  // 윈도우 메시지 포워딩
};
Template Method ▸ 메인 루프는 구체 씬을 전혀 모른 채 Init/Update/Draw/OnMessage만 호출합니다. 새 씬을 추가해도 루프 코드는 0줄 수정 — 인터페이스 기반 설계의 이점입니다.

렌더링 & RAII ENGINE

ID3DXSprite를 감싼 Sprite 클래스로 2D 출력을 추상화했습니다. GPU 텍스처 수명은 Microsoft::WRL::ComPtr(RAII)로 관리해 누수를 원천 차단했습니다.

Before — raw COM 포인터
IDirect3DTexture9* Texture;
// 소멸자에서 Release() 누락 → GPU 텍스처 누수
// 재로드 시 기존 텍스처 안 풀고 덮어씀
After — ComPtr RAII
Microsoft::WRL::ComPtr<IDirect3DTexture9> Texture;
// 소멸/재생성 시 자동 Release()
...Texture.ReleaseAndGetAddressOf());
if (FAILED(hr)) { Texture.Reset(); return false; } // 튕김 방지

렌더 API 오버로드 (용도별 분리)

함수용도
Draw(x, y)좌상단 기준 단순 출력
Draw(dx,dy, sx,sy,sw,sh)텍스처 아틀라스에서 영역(RECT) 잘라 출력
Render(x,y, radian, sx,sy, pivotMode)회전·확대·좌우반전(sx=-1), 피벗(좌상단/중앙) 선택
SetColor(r,g,b,a)틴트/페이드용 알파·컬러 변조
2D 회전의 핵심 ▸ Scale × RotationZ × Translation 행렬을 합성해 SetTransform으로 적용하고, 출력 후 반드시 단위 행렬로 복원해 다음 스프라이트에 변환이 새지 않게 합니다. 피벗을 텍스처 중앙으로 두면 제자리 회전이 가능합니다.

카메라 시스템 ENGINE

플레이어를 추적하되 맵 밖이 보이지 않도록 클램핑하고, 타격감을 위한 스크린 셰이크를 내장했습니다. 모든 렌더는 카메라 좌표를 빼서 월드→스크린 변환합니다.

📄 Camera.cpp — Update()
// 1) 타깃(플레이어)을 화면 정중앙에
camPos.posX = target->x - SCREEN_WITH  / 2.0f;
camPos.posY = target->y - SCREEN_HEIGHT / 2.0f;
// 2) 맵 경계 밖으로 못 나가게 Clamping
if (camPos.posX < 0) camPos.posX = 0;
if (camPos.posX > mapWidth - SCREEN_WITH) camPos.posX = mapWidth - SCREEN_WITH;
// 3) 셰이크: 종료 시각 전이면 랜덤 오프셋 가산 (강도 비례)
if (GetTickCount() < m_ShakeEndTime) {
  camPos.posX += ((rand()%200)-100)/100.0f * m_ShakeIntensity;
  camPos.posY += ((rand()%200)-100)/100.0f * m_ShakeIntensity;
}

CAM->Shake(20.0f, 1000) 보스 포효 시 1초간 강도 20 흔들림 · Shake(8.0f, 100) 일반 타격 시 짧고 약하게. 강도·지속을 호출부에서 조절해 이벤트별 손맛을 차등화했습니다.

충돌 / 물리 엔진 ENGINE

픽셀 단위 플랫포머 물리를 직접 구현했습니다. 핵심은 AABB 충돌 + X축·Y축 분리 해석 + 착지 좌표 보정으로 떨림·끼임 없는 이동을 만드는 것입니다.

① X축 → Y축 분리 해석

수평 이동·충돌을 먼저 끝낸 뒤 수직(중력·점프)을 따로 처리합니다. 한 번에 처리하면 모서리에서 끼이거나 벽을 타고 오르는 버그가 생기기 때문입니다.

// 수평 충돌 시 위/아래 10px 줄여 검사
wallCheckRect.top += 10; wallCheckRect.bottom -= 10;
// 바닥 1px 박힘으로 '벽'으로 오인하는 것 방지

② 착지 떨림(jitter) 제거

착지 시 캐릭터 Y를 바닥 윗면에 강제 스냅 +1.0f 보정. 매 프레임 "떨어졌다↔올라왔다"를 반복하는 진동을 없앱니다.

if (gravity >= 0 && (m_rc.bottom-30) < wall.top) {
  grounded = true; canAirDash = true; gravity = 0;
  pos.y = wall.top - (imgH-70) + 1.0f; // 떨림 방지
}

충돌 판정 — IntersectRect (AABB)

// 기사 공격박스 vs 적 몸체 → 교집합으로 타격 + 이펙트 중심 계산
if (knight.isAttacking && IntersectRect(&temp, &knight.attackBox, &e->m_rc)) {
  e->TakeDamage(1, knight.dir==1 ? -1 : 1);     // 때린 방향으로 넉백
  TIMEMGR->SetHitStop(40);  CAM->Shake(8.0f,100);  // 손맛
  float hitX = (temp.left+temp.right)/2.0f;        // 교집합 중심에 타격 이펙트
}

포고 점프(Pogo) — 게임필 디테일

TimeManager — 게임필(손맛) 엔진 ENGINE

"때리는 느낌"을 코드로 만드는 시스템. 히트스톱(타격 순간 정지)과 타임슬로우(클리어 슬로우모션)를 시간 축에서 통합 제어합니다. 싱글톤으로 어디서든 호출.

히트스톱 (Hit-Stop)

타격 순간 게임 전체를 수십 ms 멈춰 임팩트를 줍니다. 때릴 때 40ms, 맞을 때 150ms로 차등. 더 긴 정지를 짧은 것이 덮어쓰지 않도록 잔여시간을 비교하는 가드를 넣었습니다.

타임슬로우 (Time-Slow)

보스 전멸 순간 3초간 0.3배 슬로우모션. TimeAccumulator에 배율을 누적해 1.0을 넘을 때만 한 틱 진행시키는 방식으로 "부분 속도"를 구현했습니다.

bool IsTimeStopped(){
  m_TimeAccumulator += m_GlobalTimeScale; // 0.3씩
  if(m_TimeAccumulator < 1.0f) return true;
  m_TimeAccumulator -= 1.0f; return false;
}
게임 루프와의 연결 ▸ Game::Update는 매 프레임 TIMEMGR->IsTimeStopped()를 물어보고, 멈춤이면 knight/map/effect 등 "시간의 영향을 받는 오브젝트"의 Update만 건너뜁니다. UI나 입력은 그대로 도므로 정지 중에도 게임이 얼어붙지 않습니다 — 이 분리가 핵심 설계 포인트입니다.

그래프 탐색의 응용

큐 기반 확장은 BFS의 변형입니다. 단, 큐에서 무작위 인덱스를 뽑아 매번 다른 형태가 나오게 했습니다 — 랜덤 워크 + 충돌검사 조합.

이중 안전장치

확장성

왜 "동적" 벽인가

맵 구조가 매번 달라지므로 벽을 미리 박아둘 수 없습니다. 진입 시점에 m_DynamicWalls를 비우고 그 방의 형태에 맞춰 다시 생성 → 같은 프리팹이라도 이웃 연결에 따라 벽이 달라집니다.

이음매 없는 방 이동

문을 통과하면 들어간 높이/폭을 그대로 보존해 반대편에서 튀어나옵니다. 2×2 대형 방에서도 어느 셀로 들어왔는지 계산(offsetX/Y)해 좌표가 어긋나지 않습니다.

// 오른쪽 문 통과 시
knight.pos.x = 50.0f;
knight.pos.y = (gy-new_rY)*SCREEN_HEIGHT + offsetY;
특수 케이스 처리 ▸ 보스방(39번)과 보스 대기실은 그리드 바깥 규격이라 자동 캡핑 대신 통짜 사방 벽을 세웁니다. 예외를 명시적으로 분기해 일반 로직의 단순함을 유지했습니다.

★ 시체 레지스트리 — 상태 영속성 CORE LOGIC

"방을 나갔다 돌아오면 죽인 몹이 되살아나는" 흔한 문제를 해결했습니다. 각 몬스터에 결정론적 고유 ID를 부여하고, 사망 사실을 std::map에 영속화합니다.

  g->Init(info.x, info.y);  g->SetCorpse();  // 죽었던 그 자리에 시체로

왜 map 인가

map<int, CorpseInfo>는 ID로 O(log n) 조회. 죽은 몹만 저장하므로 메모리도 절약. "키를 어떻게 설계해 상태를 재현하는가"는 클라 개발의 흔한 과제입니다.

버그를 코드로 막은 흔적

SetCorpse()에서 velocity/gravity를 강제 0으로 초기화합니다 — 쓰레기 값이 남으면 시체가 화면 밖으로 날아가는 버그를 직접 디버깅하며 잡은 흔적(주석으로 명시)입니다.

하이라이트 — 예측 포물선 바운스(ROLL_BOUNCE)

단순히 위로 점프하는 게 아니라, 플레이어의 현재 위치를 향한 포물선을 계산해 날아갑니다. 체공 시간(약 50프레임)으로 수평 거리를 나눠 필요한 초기 속도를 역산하는 탄도 예측입니다.

  if (bounceCount>=3) ChangeState(IDLE);

다중 보스 페이즈 시스템 CORE LOGIC

보스방엔 3형제가 있습니다(1은 깨어남, 2·3은 SLEEP). 매니저가 매 프레임 보스들의 체력과 생존 여부를 감시하며 페이즈를 진행시킵니다 — 일종의 상위 FSM.

ComPtr·unique_ptr·CRTP 싱글톤·가상 소멸자로 누수와 초기화 순서 문제를 의식적으로 제거. 모던 C++ 감각을 보유.

예상 질문 & 답변 INTERVIEW

면접에서 나올 법한 질문과, 이 코드를 근거로 한 답변 스크립트입니다.

Q. 게임 루프에서 로직과 렌더링을 왜 분리했나요?

물리/로직을 60틱 고정으로 돌리면 PC 사양과 무관하게 같은 속도가 보장됩니다. 렌더는 가능한 만큼 빠르게 돌리되, 밀린 틱은 최대 5회까지만 따라잡게(MAX_FRAMESKIP) 해 처리가 더 느려지는 "죽음의 나선"을 막았습니다. 가변 델타타임을 그대로 물리에 쓰면 터널링·불안정이 생기기 때문에 고정 스텝을 택했습니다.

Q. 맵 생성이 무한 루프에 빠지거나 보스가 안 생기면요?

이중 안전장치를 뒀습니다. 내부 확장 루프는 failCount<1000·방 개수 32개로 종료를 보장하고, 전체 생성은 attempts<50회까지 재시도하며 보스가 배치될 때까지만 반복합니다. 또 보스 자리에 이미 방이 있으면 덮어쓰지 않는 가드가 있어, 최악의 경우에도 깨진 맵이 나오지 않습니다.

Q. 싱글톤은 안티패턴이라는데 왜 썼나요?

전역 상태 남용의 위험은 인지하고 있습니다. 다만 Camera·Sound·TimeManager처럼 게임에 단 하나만 존재하고 어디서든 접근해야 하는 시스템에 한정해 적용했습니다. new 누수와 초기화 순서 문제를 피하려고 Meyers 싱글톤(지역 static)을 CRTP로 통일했고, 복사/이동을 delete해 단일성을 강제했습니다. 규모가 커지면 의존성 주입으로 전환할 여지를 둔 설계입니다.

Q. 보스 AI가 단순 추적과 뭐가 다른가요?

IDLE에서 플레이어와의 거리를 재고 확률 테이블로 패턴을 고르는 FSM입니다. 특히 공중 바운스는 거리 ÷ 체공프레임으로 필요한 수평속도를 역산해 플레이어 위치를 조준하고, 착지할 때마다 재조준하며 벽에선 반사됩니다. 즉 "쫓아온다"가 아니라 "도착 지점을 예측해 던진다"에 가깝습니다.

Q. 방을 다시 가도 몹이 안 살아나는 건 어떻게 구현했나요?

각 몹에 방번호×1000+순서의 결정론적 ID를 줍니다. 같은 방은 항상 같은 ID를 재현하죠. 방을 떠날 때 죽은 몹의 ID·좌표를 std::map(시체 명부)에 저장하고, 들어올 때 명부에 있으면 시체 상태로 스폰합니다. 키 설계로 상태 영속성을 푼 사례입니다.

Q. 메모리 누수는 어떻게 관리했나요?

GPU 텍스처는 ComPtr로 RAII 관리해 소멸·재생성 시 자동 Release되게 했고, 씬은 unique_ptr 소유에 가상 소멸자를 둬 기반 포인터 해제가 안전합니다. 맵 이동 시엔 이전 방의 Enemy*들을 delete 후 clear하고, 이펙트는 풀에서 재사용합니다. "수명을 누가 소유하는가"를 타입으로 드러내려 했습니다.
