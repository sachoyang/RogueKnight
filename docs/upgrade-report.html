✅ 적용됨 CRIT 🐞 버그/UB 베이스 클래스 가상 소멸자 추가 + 씬 unique_ptr 소유 Chap.h · CMng.h · Main.cpp ▶

문제: 씬을 Chap*로 다형 처리하는데 소멸자가 virtual이 아니라 delete 시 UB, 게다가 new한 씬을 한 번도 해제하지 않음. 조치: 가상 소멸자 + std::unique_ptr 소유로 종료 시 자동 해제.

❌ Before

    ~Chap() {};            // virtual 아님
};
Chap* chap[TOTALCHAP];      // raw
chap[LOGO] = new Logo;     // 해제 없음
✅ After
class Chap {
    virtual ~Chap() = default;   // ★ 가상 소멸자
};
std::unique_ptr<Chap> chap[TOTALCHAP];
chap[LOGO] = std::make_unique<Logo>();
// 종료 시 자동 해제

✅ 적용됨 HIGH 🏛️ 구조 싱글톤 패턴 통일 — CRTP Singleton<T> 베이스 Singleton.h · Camera/Sound/Effect/Time ▶

문제: Camera는 new+전역 포인터, Sound/Effect/Time은 각자 함수내 static 으로 싱글톤을 제각각 구현. 조치: Singleton<T> CRTP 베이스 신설 → 4개 매니저가 모두 상속(복사/이동 금지, 누수·초기화순서 문제 해소). 기존 CAM->, SOUND-> 호출부는 그대로 유지.

❌ Before
// Camera.cpp — new 기반 + 누수
Camera* Camera::m_pInstance = nullptr;
Camera* Camera::GetInstance(){
  if(!m_pInstance) m_pInstance=new Camera();
  return m_pInstance;
}
// Sound/Effect/Time — 또 다른 방식
static Sound* GetInstance(){static Sound i;return &i;}
✅ After
// Singleton.h — 단일 베이스
template<class T> class Singleton {
public:
  static T* GetInstance(){static T i;return &i;}
  Singleton(const Singleton&)=delete;
  Singleton& operator=(const Singleton&)=delete;
protected: Singleton()=default;
};
class Camera : public Singleton<Camera> { ... };
✅ 적용됨 HIGH 🧹 메모리/RAII Sprite 텍스처 RAII화 (ComPtr) — GPU 누수 차단 Sprite.h · Sprite.cpp ▶

문제: IDirect3DTexture9*를 들고 있으나 소멸자에서 Release()하지 않아 GPU 리소스 누수, 재생성 시에도 누적. 조치: Microsoft::WRL::ComPtr로 감싸 소멸·재생성 시 자동 Release. 호출부는 .Get()/.Reset()로 안전화.

❌ Before
IDirect3DTexture9* Texture;       // raw
~Sprite(){}                        // Release 없음
D3DXCreateTextureFromFileEx(...,&Texture);
if(Texture==NULL) return;
Sprite->Draw(Texture, ...);
✅ After
Microsoft::WRL::ComPtr<IDirect3DTexture9> Texture;
// 소멸자 불필요 — 자동 Release
D3DXCreateTextureFromFileEx(...,
    Texture.ReleaseAndGetAddressOf());
if(!Texture) return;
Sprite->Draw(Texture.Get(), ...);
✅ 적용됨 MED ✨ 모던 C++ #define 상수 → constexpr (타입·스코프 안전) Define.h ▶

❌ Before

#define RIGHT 1
✅ After

🔵 향후 HIGH 🏛️ 구조 전역 extern 객체 → 싱글톤 통합 (knight·coll·g_Mng 등) 전역 전반 ▶

왜 미루었나(의도적): knight(144회)·coll(47)·g_Mng(51) 등 호출처가 매우 많고, 전역 객체는 프로그램 시작 시점에 생성되는 반면 싱글톤은 최초 접근 시(lazy) 생성되어 초기화 타이밍이 바뀝니다(예: GameManager 생성자가 파일 IO 수행). 런타임 회귀 위험이 있어 별도 단계에서 진행하는 것이 안전합니다.

class Knight : public Singleton<Knight> {...};

🔵 향후 MED ✨ 모던 C++ C 파일 I/O(fopen/itoa) → fstream / to_string GameManager.cpp ▶

제안: 세이브 로직의 fopen/fwrite/fclose를 RAII std::ofstream으로, itoa를 std::to_string으로. 현재 세이브/로드 경로가 일부 주석 상태라 기능 정리와 함께 진행 예정.

🧩 디자인 패턴 로드맵 적용된 패턴과, 포트폴리오 가치를 더 높일 다음 단계

Singleton (CRTP) 매니저 접근 통일 ✅ 적용됨

Singleton<T> 베이스로 Camera/Sound/EffectManager/TimeManager를 통일. 복사·이동을 delete로 막고, Meyers 싱글톤으로 초기화 순서 문제와 new 누수를 동시에 제거.

RAII 리소스 수명 자동화 ✅ 적용됨

씬은 unique_ptr, GPU 텍스처는 ComPtr로 소유. 소멸·예외·조기 return 어디서든 자동 해제되어 누수가 구조적으로 사라짐.

State 씬·게임 상태 머신 🔵 향후

이미 Chap(씬) 추상화가 State 패턴의 씨앗. SceneManager(전환/스택)로 정식화하고 Playing/Paused/Cutscene를 명시적 상태로 표현하면 'p키 vs ESC' 같은 상태 꼬임이 구조적으로 해결.

Object Pool 이펙트·적·투사체 풀링 🔵 향후

EffectManager가 타격마다 std::list에 생성/삭제 → 전투 중 잦은 할당이 끊김 유발. 고정 크기 풀에서 꺼내 쓰고 반납하면 할당 비용·메모리 변동 제거.

Component / Command 엔티티·입력 추상화 🔵 향후

Knight/Enemy/Boss의 공통 필드(위치·HP·히트박스·넉백)를 Entity 베이스로 추출. 흩어진 GetAsyncKeyState 입력을 "점프/대시/공격" Command로 추상화하면 키 리매핑·리플레이·테스트가 가능.
